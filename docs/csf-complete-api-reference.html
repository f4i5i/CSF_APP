1. Global: Authentication, Errors & Pagination

Authentication

All protected endpoints require a JWT Bearer token:

Authorization: Bearer {access_token}
Content-Type: application/json

Tokens are obtained via login and stored locally. On 401 response, redirect to login. Tokens can be refreshed:

MethodEndpointBodyResponse
POST/auth/login{ email, password }{ access_token, refresh_token, token_type, user, must_change_password }
POST/auth/register{ email, password, first_name, last_name, phone }{ access_token, refresh_token, user }
POST/auth/refresh{ refresh_token }{ access_token, refresh_token }
POST/auth/google{ token } (Google OAuth token){ access_token, refresh_token, user }
POST/auth/logoutNoneConfirmation
POST/auth/forgot-password{ email }Confirmation message
POST/auth/reset-password{ token, new_password }Confirmation
POST/auth/change-password{ current_password, new_password }Confirmation
Force Password Change: If must_change_password: true in login response, redirect user to force password change screen before allowing any other action.
Role Mapping: Login response has a user.role field. Both OWNER and ADMIN roles are stored as "admin" locally. Other roles: "parent", "coach".

Error Handling

// Standard error response
{
  "detail": "Error message string"
}

// Validation error response
{
  "detail": [
    { "loc": ["body", "field_name"], "msg": "error message", "type": "error_type" }
  ]
}

4. Parent Pages

4.1 Parent Dashboard PARENT

Route: /dashboard

Main parent landing page showing overview of all children's activities.

UI Layout: Welcome header with child+class dropdown selector at top. Below that: a waiver alert banner (if unsigned waivers exist), two stat cards (Attendance Streak, Badges Earned), an announcements list on the left, and a widgets panel on the right (mini calendar, next event, recent photo, badge showcase).

APIs on page load (parallel):
GET /enrollments/my?child_id={id}&status=active - Populates the child+class dropdown selector at top. Groups enrollments by child. Each enrollment provides class_id used to fetch all subsequent data. Also used to calculate "Total Active Enrollments" stat.
GET /waivers/pending - Yellow alert banner at top of page: "You have X unsigned waivers". Shows count and links to /waivers page. Hidden if no pending waivers.
APIs after enrollment loads (depend on selected child+class):
GET /announcements?class_id={id} - Left panel: "Announcements" card showing coach announcements for the selected class. Displays title, content preview, author name, and timestamp. Limited to most recent ~5.
GET /badges/enrollment/{enrollmentId} - Right panel: "Badges" widget showing the 4 most recently earned badges with icons, names, and earned dates. Also used for "Badges Earned" stat counter at the top.
GET /attendance/enrollment/{enrollmentId}/streak - Top stat card: "Attendance Streak" showing current_streak number (e.g., "5 day streak"). No detailed breakdown on dashboard - just the number.
GET /photos/class/{classId}?limit=6 - Right panel: "Recent Photo" widget showing the first photo thumbnail from the class gallery. Tapping navigates to full gallery page.
GET /events/class/{classId} - Right panel: Mini calendar with event dots + "Next Event" card showing the soonest upcoming event with title, date, time, and location.
GET /events?upcoming=true&limit=5 - Fallback only: loaded instead of class events when no enrollment/class is selected. Shows general upcoming events.

4.3 Checkout & Payment Flow PARENT

Routes: /checkout, /payment/success, /payment/cancel

UI Layout: Multi-step checkout page with: class details summary at top, child selector (supports multi-child enrollment), optional custom fees checkboxes, discount code input, payment method selector (saved cards or new card), optional installment plan toggle, order summary with line items, and a "Pay Now" button. A waiver modal may appear before payment if unsigned waivers exist.

The checkout flow involves multiple sequential API calls driven by the useCheckoutFlow hook:

Step 1 - Page initialization (parallel):
GET /classes/{classId} - Top section: "Class Details Summary" card showing class name, schedule, coach, school, price, age range. The classId comes from URL query param ?classId=xxx.
GET /children/my - "Select Child" section: shows parent's children as selectable cards. Supports multi-child selection for enrolling siblings. Each card shows child name, age, grade.
GET /classes/{classId}/capacity - If class is full (hasCapacity: false), shows "Join Waitlist" button instead of checkout form. Capacity check happens silently.
Step 2 - User applies discount (optional, on button click):
POST /discounts/validate - "Discount Code" input field with "Apply" button. On success, shows green badge with discount amount/percentage and updates order total in real-time. "Remove" button clears the discount.
Step 3 - Before payment (on "Pay Now" click):
GET /waivers/pending - Checks for unsigned waivers. If found, a modal overlay ("WaiverCheckModal") appears listing each waiver with checkbox and signature field. Parent must sign all waivers before payment proceeds.
POST /waivers/accept - Called for each waiver signed in the modal (looped sequentially).
Step 4 - Payment processing (sequential):
POST /orders - Creates the order with selected children, class, discount code, and custom fees. Returns order total, line items breakdown, and order ID.
POST /orders/{orderId}/pay - Creates Stripe PaymentIntent. Returns client_secret for Stripe SDK. If user has a saved card, payment_method_id is sent.
[Stripe SDK handles card input on client side]
POST /payments/{paymentIntentId}/confirm - Called after Stripe confirms payment. Finalizes the enrollment. On success, shows "Order Confirmation" screen with receipt download button.

4.4 Payments & Billing PARENT

Route: /paymentbilling

UI Layout: Settings-style page with left sidebar navigation and 4 vertically stacked content sections on the right: Payment Methods, Billing, Memberships, and Invoices. On mobile, sections stack with section headers between them.

APIs on page load (4 parallel, independent calls):
GET /payments/methods - Section 1 "Payment Methods": shows saved credit/debit cards in a list. Each card shows brand logo (Visa/MC/Amex), last 4 digits, expiry date, and "Default" badge. Includes "Add Card" button and per-card "Set Default" / "Remove" actions.
GET /invoices/billing-summary - Section 2 "Billing": shows billing period dates, "Next Payment Due" card with amount and date (highlighted yellow), active enrollments count, and pending payments warning banner.
GET /enrollments/my - Section 3 "Memberships": shows all enrollments grouped by child (collapsible child headers). Each enrollment row shows class name, status badge, weekdays, enrolled date, price, and action buttons (Pause/Resume/Cancel). Bottom summary shows total active count and total monthly cost.
GET /invoices/my - Section 4 "Invoices": sortable table with columns: Invoice #, Date, Description, Amount, Status badge, Download button. Filter buttons at top (All/Paid/Pending/Overdue). "Sync from Stripe" refresh button.

4.5 My Children PARENT

Routes: /mychildren, /editchild/:childId, /addchild

UI Layout: /mychildren shows a list/grid of child profile cards. Each card displays child name, age, grade, school, profile photo, and action buttons (Edit, Delete). /editchild/:childId shows a full form pre-filled with the child's data. /addchild is a simple form for quick child creation.

APIs:
GET /children/my - Main content: loads all parent's children as profile cards. Each card shows: profile photo (or avatar placeholder), full name, date of birth/age, grade level, school, and emergency contact summary.
POST /children - "Add Child" button opens form. Sends child profile data including emergency contacts array (required, min 1). On success, refreshes the children list.
GET /children/{id} - Edit Child page: pre-fills the edit form with all child data (name, DOB, grade, medical info, emergency contacts).
PUT /children/{id} - "Save" button on edit form. Sends updated child data. Note: emergency contacts are NOT updated via this endpoint (they have separate endpoints).
GET /children/{id}/delete-check - Called when user taps "Delete" button. Returns whether deletion is safe. If child has active enrollments, shows warning: "Child has X active enrollments that must be cancelled first" and blocks deletion.
DELETE /children/{id} - Called after delete-check passes and user confirms. Soft-deletes the child. Refreshes children list.

4.6 Waivers PARENT

Route: /waivers

UI Layout: Full-screen waiver signing page with CSF branding. Shows a card with: logo + "Registration Waiver" title, then a scrollable list of each pending waiver (title, full legal text, checkbox "I agree"), a typed signature input field at the bottom, and Submit/Cancel buttons.

APIs:
GET /waivers/pending - Page load: fetches all unsigned waivers. Each waiver is rendered as a bordered section with the waiver name as header, full content/legal text below, and a checkbox "I agree to the [name] terms". If response returns empty array, shows blue info box: "No pending waivers found - All required waivers have been signed".
POST /waivers/accept - "Submit Waiver" button: loops through all checked waivers and calls this endpoint for each one sequentially. Sends template_id, signature (typed name), and agreed: true. On success, shows toast and redirects to /dashboard after 1.5s.

POST /waivers/accept

8. Stripe Integration Notes

  • Adding cards uses Stripe's SetupIntent flow (PCI compliant)
  • Mobile apps must use the Stripe Mobile SDK (iOS: stripe-ios, Android: stripe-android) instead of Stripe.js
  • The client_secret from POST /payments/setup-intent is passed to the Stripe SDK to collect card details
  • Card details NEVER pass through the CSF backend - they go directly to Stripe
  • The Stripe publishable key is: REACT_APP_STRIPE_PUBLISHABLE_KEY (get from env config)
  • Payment flow: POST /orders/{id}/pay returns a client_secret for Stripe PaymentIntent
  • After Stripe confirms payment on client side, call POST /payments/{paymentIntentId}/confirm
  • Webhooks: Stripe sends events to POST /webhooks/stripe (server-side only)
