Parent Dashboard - Payments & Billing Page: API Reference

For: Mobile Developer
Route: /paymentbilling (protected, parent role only)
Base URL: {API_BASE_URL} (all endpoints are relative to this)
Auth: Bearer token (JWT) in Authorization header

Page Overview

The Payments & Billing page has 4 sections, each powered by different API endpoints:

#SectionComponentDescription
1Payment MethodsPaymentCardSaved cards (Stripe) - add, remove, set default
2Billing SummaryBillingInfoNext due date, active enrollments count, pending payments
3MembershipsMembershipListActive enrollments with cancel/pause/resume actions
4InvoicesInvoiceTableInvoice list with filtering, download PDF, Stripe sync

Section 1: Payment Methods (Stripe Integration)

1.1 Get Saved Payment Methods

GET /payments/methods

Response: Array of payment methods

[
  {
    "id": "pm_1234567890",
    "brand": "visa",
    "last4": "4242",
    "exp_month": 12,
    "exp_year": 2027,
    "is_default": true
  }
]
Note: Response may be { items: [...] } or a plain array. Handle both formats.

1.2 Create Setup Intent (for adding new card)

POST /payments/setup-intent

Request Body: None

Response:

{
  "client_secret": "seti_xxx_secret_xxx"
}
Stripe Flow: Use the client_secret with Stripe SDK's confirmCardSetup() to securely collect card details. After Stripe confirms, the card is automatically saved. Mobile apps should use the Stripe Mobile SDK (iOS/Android) instead of Stripe.js.

1.3 Set Default Payment Method

POST /payments/methods/{methodId}/default

Path Params: methodId - Stripe payment method ID (e.g., pm_xxx)

Request Body: None

Response: Confirmation object

1.4 Remove Payment Method

DELETE /payments/methods/{methodId}

Path Params: methodId - Stripe payment method ID

Response: Confirmation object

Section 2: Billing Summary

2.1 Get Billing Summary

GET /invoices/billing-summary

Response:

{
  "billing_period_start": "2026-03-01",
  "billing_period_end": "2026-03-31",
  "next_due_date": "2026-04-01",
  "next_payment_amount": 150.00,
  "active_enrollments_count": 3,
  "pending_payments_count": 1
}

UI Logic:

  • Show "Next Payment Due" card only if both next_due_date AND next_payment_amount exist
  • Show "Active Enrollments" count (default to 0 if null)
  • Show pending payments warning banner if pending_payments_count > 0

Section 3: Memberships (Enrollments)

3.1 Get My Enrollments

GET /enrollments/my

Optional Query Params:

ParamTypeDescription
statusstringFilter: ACTIVE, PENDING, COMPLETED, CANCELLED, WAITLISTED, PAUSED
child_idstring (UUID)Filter by child
class_idstring (UUID)Filter by class

Response: Array of enrollments (may also be { items: [...] })

[
  {
    "id": "uuid",
    "child_id": "uuid",
    "child_name": "John Doe",
    "class_name": "Soccer Basics",
    "status": "ACTIVE",
    "final_price": 120.00,
    "discount_amount": 10.00,
    "payment_completed": true,
    "enrolled_at": "2026-01-15T10:00:00Z",
    "created_at": "2026-01-14T08:00:00Z",
    "weekdays": ["Monday", "Wednesday"]
  }
]

UI Logic:

  • Group enrollments by child_id to show per-child sections
  • Use child_name (flat string) or fallback to child.first_name + child.last_name
  • Use class_name or fallback to class.name
  • Show "Pause" button only for ACTIVE status
  • Show "Resume" button only for PAUSED status
  • Show "Cancel" button for: ACTIVE, PENDING, WAITLIST, PAUSED, WAITLISTED
  • Calculate total monthly cost from all ACTIVE enrollments' final_price

Status badge colors:

StatusColor
ACTIVEGreen
PENDINGYellow
COMPLETEDBlue
CANCELLEDGray
WAITLISTEDPurple
PAUSEDAmber

3.2 Get Cancellation Preview

GET /enrollments/{enrollmentId}/cancellation-preview

Path Params: enrollmentId - UUID

Response:

{
  "refund_amount": 120.00,
  "cancellation_fee": 15.00,
  "net_refund": 105.00,
  "refund_policy": "Prorated refund based on remaining classes"
}
Usage: Call this when user taps "Cancel" to show refund details in a confirmation dialog BEFORE actually cancelling.

3.3 Cancel Enrollment

POST /enrollments/{enrollmentId}/cancel

Request Body:

{
  "reason": "Schedule conflict",
  "refund_requested": true
}

Response:

{
  "message": "Enrollment cancelled successfully",
  "refund_amount": 105.00
}

3.4 Pause Enrollment

POST /enrollments/{enrollmentId}/pause

Request Body:

{
  "reason": "Family vacation"
}
reason is optional.

Response: Updated enrollment object with status: "PAUSED"

3.5 Resume Enrollment

POST /enrollments/{enrollmentId}/resume

Request Body: None

Response: Updated enrollment object with status: "ACTIVE"

4.2 Download Invoice PDF

GET /invoices/{invoiceId}/download

Path Params: invoiceId - UUID

Response: Binary PDF blob (Content-Type: application/pdf)

Save/share the PDF file using the platform's native file handling.

4.3 Sync Invoices from Stripe

POST /invoices/sync

Request Body: None

Response:

{
  "summary": {
    "created": 2,
    "updated": 5
  }
}
Fetches latest invoices from Stripe and syncs to database. Call this when user taps a "Refresh" button.

API Calls on Page Load

When the page loads, these 4 APIs are called simultaneously (all independent, can be parallel):

1. GET /payments/methods          → Payment Methods section
2. GET /invoices/billing-summary  → Billing Summary section
3. GET /enrollments/my            → Memberships section
4. GET /invoices/my               → Invoices section

Authentication

All requests require:

Headers:
  Authorization: Bearer {jwt_token}
  Content-Type: application/json

The JWT token is obtained from the login endpoint and stored locally. If a request returns 401 Unauthorized, redirect to login.

Error Handling

Standard error response format:

{
  "detail": "Error message describing what went wrong"
}
HTTP StatusMeaning
400Bad request / validation error
401Not authenticated (redirect to login)
403Not authorized for this action
404Resource not found
409Conflict (e.g., duplicate enrollment)
500Server error

Stripe Integration Notes

  • Adding cards uses Stripe's SetupIntent flow (PCI compliant)
  • Mobile apps should use the Stripe Mobile SDK (iOS/Android) instead of Stripe.js
  • The client_secret from /payments/setup-intent is used with Stripe SDK to collect card details
  • Card details NEVER pass through the backend - they go directly to Stripe
  • The Stripe publishable key is configured via environment variable: REACT_APP_STRIPE_PUBLISHABLE_KEY
